How Memory Works in Vanilla OpenClaw

Three different backends, three different philosophies

memory-core (default)

Storage: Markdown files (MEMORY.md + daily notes)

Tools: memory_search, memory_get, pre-compaction flush

Pros: Human-readable, simple, works offline

Cons: No semantic search, linear lookup, manual curation needed

memory-lancedb (plugin)

Storage: LanceDB vector store with autoCapture/autoRecall

Requirements: OpenAI embeddings API (costs money)

Pros: True semantic search, automatic capture

Cons: Requires API key, not self-hosted, vendor lock-in

QMD (experimental)

Architecture: BM25 + vectors + reranking sidecar

Installation: Separate CLI install, external daemon

Pros: Hybrid search (keyword + semantic)

Cons: Complex setup, maintenance overhead, experimental status

The Problem

Each approach solves part of the puzzle, but none deliver a complete solution. Users are forced to choose between simplicity (markdown), vendor dependency (LanceDB), or complexity (QMD). Meanwhile, the community has been requesting unified memory for 2+ years...

The Uncomfortable Truth

Our 946-line memory-unified plugin does more actual work than Ruflo's entire 6006-line memory subsystem + 1447-line neural module + 1267-line memory CLI. Because our plugin connects to real services (Qwen3 on Spark, better-sqlite3 native bindings) while Ruflo's code talks to itself through hash embeddings.

The value was in the CONCEPTS, not the implementation.

Implementation Roadmap

4 phases, ~8.5 hours total effort

P0

Phase 0: Native HNSW with Real Embeddings

~3h

Replace Ruflo MCP dependency with native HNSW in memory-unified.

  • Add hnswlib-node dependency (4096-dim, cosine metric)
  • after_tool_call hook: embed via Qwen3 → add to HNSW
  • before_agent_start hook: query HNSW alongside FTS5
  • Persist to skill-memory.hnsw (auto-save/load)
  • Migrate existing Ruflo data with real embeddings
Success metric: curl localhost:3002/mcp returns 404 and nothing breaks
P1

Phase 1: Pattern Learning with Temporal Decay

~2h

Extract patterns from skill execution history with confidence scoring.

  • Add patterns + pattern_history tables (Ruflo schema)
  • agent_end hook: extract pattern from (skill, keywords, success/failure)
  • Daily cron: decay confidence of unused patterns
  • Boost FTS5 results with pattern confidence
  • unified_patterns tool for inspection
Success metric: After 1 week: >20 patterns with meaningful confidence scores
P1

Phase 2: Data Migration + Memory Classification

~2h

Richer memory taxonomy + migrate valuable Ruflo data.

  • Export trading (1212) + knowledge-base (127) from Ruflo
  • Re-embed with Qwen3 (real 4096-dim vs hash fallback)
  • Add memory_type column (semantic/episodic/procedural/working/pattern)
  • Add access_count, last_accessed_at (hot/cold tracking)
  • Auto-classify on insert
Success metric: 1339 entries migrated with real embeddings
P2

Phase 3: Ruflo Deprecation + Cleanup

~1h

Remove unnecessary dependencies and clean up legacy code.

  • Remove RUFLO_MCP_URL from openclaw.json
  • Stop mcp-http-proxy.js process
  • Archive /home/tank/ruflo-workspace/ (reference only)
  • Clean up 990 old skill entries
  • Update documentation
Success metric: Zero Ruflo processes, faster gateway startup
